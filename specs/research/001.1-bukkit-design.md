# Story 001.1: BUKKIT Syntax Research & Design

**Status**: ✅ Complete  
**Date**: 2026-05-15  
**Epic**: EPIC-001 Arrays and Dictionaries

## Executive Summary

This document presents comprehensive research on BUKKIT (array/dictionary) implementations in LOLCODE and defines the design decisions for LULCODE's BUKKIT support. The key decision is to adopt a **Lua-table-style hybrid data structure** that supports both integer indexing and string keys, with **bracket notation sugar** (`arr[key]`) that transpiles to LOLCODE's `'Z` syntax.

---

## Research Findings

### 1. GitHub Issue #11 (2018 Community Proposal)

**Source**: https://github.com/justinmeza/lolcode-spec/issues/11

The original BUKKIT proposal from the LOLCODE spec repository was minimal:

**Proposed Syntax**:
```lolcode
I HAS A TEST ITZ []
PUT "hello" IN TEST
```

**Limitations**:
- No access syntax defined ("todo: accessing contents")
- Limited community discussion or feedback
- Never incorporated into official spec
- Abandoned or incomplete proposal

**Analysis**: Too minimal for practical use. The `[]` syntax breaks LOLCODE's verbose style, and `PUT ... IN` doesn't distinguish between append vs indexed assignment.

### 2. LCI Implementation (Current De Facto Standard)

**Sources**: 
- https://www.tutorialspoint.com/lolcode/lolcode_types.htm
- https://github.com/BGP100/LOLCODE-Guide/blob/main/Types.md
- https://github.com/leyarotheconquerer/lolcode-reference/blob/master/AdvancedReference.md

LCI (LOLCODE Interpreter in C by Justin Meza) provides the most complete BUKKIT implementation:

#### Creation Syntax
```lolcode
I HAS A array ITZ A BUKKIT
```

#### Property-Style Access (Named Keys)
```lolcode
BTW Create properties
array HAS A keyname ITZ "value"

BTW Access properties
VISIBLE array'Z keyname    BTW Output: value

BTW Update properties
array'Z keyname R "new value"
```

#### Numeric Index Access (Array Style)
```lolcode
BTW Set numeric indices
array'Z SRS 0 R "first"     BTW SRS = "seriously" (array subscript)
array'Z SRS 1 R "second"

BTW Using variable index
I HAS A index ITZ 2
array'Z SRS index R "third"

BTW Read values
VISIBLE array'Z SRS 0       BTW Output: first
```

#### Function Storage
```lolcode
BTW Store function references
array'Z myFunc R functionName

BTW Call stored function
I IZ array'Z myFunc MKAY
```

#### Iteration
```lolcode
I HAS A i ITZ 0
IM IN YR loop UPPIN YR i TIL BOTH SAEM i AN LENGZ OF array
  VISIBLE array'Z SRS i
IM OUTTA YR loop
```

**Key Insight**: LCI's BUKKIT is a **hybrid structure**—it supports both named properties (like a dictionary/object) and numeric indices (like an array), similar to Lua tables or PHP arrays.

### 3. Object-Oriented Syntax (Advanced LCI Features)

Some LOLCODE implementations support OO-style BUKKIT declaration:

```lolcode
O HAI IM objectname
  I HAS A property ITZ "value"
  
  HOW IZ I method
    BTW Access own properties with ME'Z
    VISIBLE ME'Z property
  IF U SAY SO
KTHX
```

**Analysis**: This is sugar for BUKKIT with methods. Demonstrates that BUKKIT can serve as the foundation for object-oriented programming in LOLCODE.

### 4. Other Implementations

**lol-coffee** (used in lolpoker project):
- Does NOT support BUKKIT natively
- Comment in game.lol explicitly states: *"All state kept as strings (no BUKKIT support in lol-coffee)"*
- This is why game.lol uses string concatenation workarounds

**Community Implementations**:
- Various partial implementations exist
- No consistent syntax across implementations
- LCI's syntax has emerged as de facto standard

## Syntax Comparison Table

| Feature | GitHub #11 Proposal | LCI Implementation | LULCODE Decision |
|---------|--------------------|--------------------|------------------|
| **Creation** | `ITZ []` | `ITZ A BUKKIT` | `ITZ A BUKKIT` (std) + `= BUKKIT` (sugar) |
| **Named Access** | Undefined | `arr'Z key` | `arr'Z key` (std) + `arr[key]` (sugar) |
| **Numeric Access** | Undefined | `arr'Z SRS index` | `arr'Z SRS index` (std) + `arr[index]` (sugar) |
| **Assignment** | `PUT val IN arr` | `arr'Z key R val` | `arr'Z key R val` (std) + `arr[key] = val` (sugar) |
| **Property Creation** | Undefined | `arr HAS A key ITZ val` | `arr HAS A key ITZ val` (standard) |
| **Iteration** | Undefined | `IM IN YR` with SRS | `IM IN YR` (std) + `FOREACH` (sugar) |
| **Semantics** | Unclear | Hybrid (array + dict) | **Hybrid** (follows LCI) |

## Design Decisions

### ✅ Decision 1: BUKKIT Semantics (Hybrid Array/Dictionary)

**Choice**: Adopt **Lua-table-style hybrid semantics** supporting both integer-indexed arrays and string-keyed dictionaries in a single data structure.

**Rationale**:
1. **Proven Implementation**: Matches LCI's working implementation
2. **Maximum Flexibility**: Users don't need separate `ARRAY` and `DICT` types
3. **Minimalist Philosophy**: Aligns with Lua's "one universal data structure" principle
4. **Real-World Need**: game.lol needs dictionary behavior for VOTES tracking
5. **Familiar Model**: Similar to Lua tables, PHP arrays, JavaScript objects

**Examples**:
```lulcode
I HAS A mixed ITZ A BUKKIT

BTW Use as array
mixed'Z SRS 0 R "first"
mixed'Z SRS 1 R "second"

BTW Use as dictionary
mixed'Z name R "Alice"
mixed'Z score R 100

BTW Mixed usage is valid
VISIBLE mixed'Z SRS 0        BTW Output: first
VISIBLE mixed'Z name         BTW Output: Alice
```

### ✅ Decision 2: BUKKIT Creation Syntax

**Choice**: `I HAS A name ITZ A BUKKIT` (LCI standard)

**Alternatives Considered**:
- ❌ `I HAS A name ITZ []` (Issue #11) - Too terse, breaks LOLCODE style
- ❌ `I HAS A name ITZ BUKKIT` (no A) - Grammatically inconsistent with LOLCODE
- ✅ Allow both standard and LULCODE sugar (EPIC-003)

**LULCODE Enhancement** (future, EPIC-003):
```lulcode
I HAS A votes ITZ A BUKKIT   BTW Standard LOLCODE (always works)
VAR votes = BUKKIT           BTW LULCODE syntactic sugar
```

**Backward Compatibility**: Full - uses official LOLCODE syntax

### ✅ Decision 3: BUKKIT Access Syntax

**Choice**: Support **multiple access patterns** with progressive enhancement

#### Standard LOLCODE Syntax (for transpilation target):

**Named Keys** (string literals as identifiers):
```lolcode
votes'Z Alice    BTW Access key "Alice"
```

**Numeric Keys** (integer indices):
```lolcode
votes'Z SRS 0    BTW Access index 0 (SRS = "seriously")
votes'Z SRS index BTW Access index stored in variable
```

#### LULCODE Sugar (transpiles to above):

**Bracket Notation**:
```lulcode
votes["Alice"]   BTW → votes'Z Alice
votes[0]         BTW → votes'Z SRS 0
votes[index]     BTW → votes'Z SRS index
```

**Reasoning**:
- **LCI's `'Z` syntax is cryptic**: Hard to read, non-obvious for new users
- **Bracket notation is universal**: Every modern language uses `arr[key]`
- **Easy transpilation**: LULCODE parser can detect string vs numeric keys
- **Backward compatible**: Both syntaxes work; pure LOLCODE files are valid LULCODE

**Transpilation Rules**:
1. `arr[stringLiteral]` → `arr'Z stringLiteral` (no quotes)
2. `arr[identifier]` → `arr'Z SRS identifier` (variable lookup)
3. `arr[numericExpr]` → `arr'Z SRS (numericExpr)` (evaluate index)

### ✅ Decision 4: BUKKIT Assignment Syntax

**Choice**: Support both **property creation** and **direct assignment**

#### Method 1: Property Creation (adding new keys)
```lolcode
bukkit HAS A keyname ITZ value
```
- LOLCODE's standard syntax for creating new properties
- Clear intent: adding a new key-value pair
- Works with LCI

#### Method 2: Direct Assignment (updating existing keys)
```lolcode
bukkit'Z keyname R value
```
- Updates value at existing key
- Can also create new keys in LCI

#### LULCODE Sugar:
```lulcode
bukkit[key] = value    BTW Transpiles to bukkit'Z key R value
```

**Best Practice for LULCODE**:
- Use `HAS A` when first declaring keys (clearer)
- Use `[key] = val` or `'Z key R val` for updates

### ✅ Decision 5: BUKKIT Iteration

**Choice**: Enhance `IM IN YR` with BUKKIT-aware syntax

#### Standard LOLCODE (numeric iteration only):
```lolcode
I HAS A i ITZ 0
IM IN YR loop UPPIN YR i TIL BOTH SAEM i AN LENGZ OF bukkit
  VISIBLE bukkit'Z SRS i
IM OUTTA YR loop
```

**Limitation**: Can only iterate numeric indices, not arbitrary keys

#### LULCODE Enhancement (future, EPIC-003):
```lulcode
FOREACH key, value IN bukkit
  VISIBLE SMOOSH "Key: " key ", Value: " value MKAY
END
```

**Transpilation Strategy**:
1. Maintain internal key list during BUKKIT operations
2. Transpile `FOREACH` to `IM IN YR` loop over key list
3. Use `key` to access `bukkit'Z SRS key` inside loop

**Implementation Notes**:
- Requires tracking BUKKIT keys at transpile time OR runtime
- May need LULCODE runtime library for key enumeration
- Consider deferring to Story 001.4 (Built-in Operations)

### ✅ Decision 6: Built-in BUKKIT Operations

**Choice**: Prioritize essential operations, defer advanced ones

#### Essential Operations (Priority 1 - MVP):

**Length**:
```lolcode
LENGZ OF bukkit    BTW Returns number of keys
```
- Reuses existing LOLCODE `LENGZ` operator
- LCI support: TBD (may need LULCODE extension)

**Key Existence Check**:
```lolcode
bukkit HAS A keyname   BTW Returns TROOF (WIN/FAIL)
```
- Already exists for property creation, extend for checking

**Access**:
```lolcode
bukkit'Z key       BTW Standard
bukkit[key]        BTW LULCODE sugar
```

#### Nice-to-Have Operations (Priority 2 - Future):

**Delete Key**:
```lolcode
REMOVE keyname FROM bukkit
```

**Get All Keys**:
```lolcode
KEYZ OF bukkit     BTW Returns BUKKIT of keys
```

**Get All Values**:
```lolcode
VALUEZ OF bukkit   BTW Returns BUKKIT of values
```

**Merge BUKKITs**:
```lolcode
SMOOSH bukkit1 AN bukkit2 MKAY   BTW Extend SMOOSH for BUKKITs
```

**Defer to**: EPIC-002 (Standard Library) or future enhancements

## Transpilation Strategy

### Overview

LULCODE BUKKIT syntax transpiles to valid LOLCODE that can run on LCI or extended lol-coffee:

```
LULCODE (.lul) with brackets
    ↓
LULCODE Parser (detects arr[key] patterns)
    ↓
LULCODE AST (BracketAccessNode)
    ↓
Code Generator (emits arr'Z key or arr'Z SRS index)
    ↓
LOLCODE (.lol) with 'Z syntax
    ↓
LCI or lol-coffee execution
```

### Transpilation Rules

#### Rule 1: Bracket Access → 'Z Syntax

**String Literal Keys**:
```lulcode
arr["name"]        →  arr'Z name
```

**Identifier Keys**:
```lulcode
arr[keyVar]        →  arr'Z SRS keyVar
```

**Numeric Literal Keys**:
```lulcode
arr[0]             →  arr'Z SRS 0
```

**Numeric Expression Keys**:
```lulcode
arr[i + 1]         →  arr'Z SRS SUM OF i AN 1
```

#### Rule 2: Bracket Assignment → 'Z Assignment

```lulcode
arr[key] = value   →  arr'Z SRS key R value
```

#### Rule 3: FOREACH → IM IN YR

```lulcode
FOREACH k, v IN arr
  VISIBLE v
END
```

Transpiles to:
```lolcode
I HAS A __keys ITZ KEYZ OF arr
I HAS A __i ITZ 0
IM IN YR loop UPPIN YR __i TIL BOTH SAEM __i AN LENGZ OF __keys
  I HAS A k ITZ __keys'Z SRS __i
  I HAS A v ITZ arr'Z SRS k
  VISIBLE v
IM OUTTA YR loop
```

*(Note: Requires `KEYZ OF` built-in, deferred to Story 001.4)*

### Implementation Phases

**Phase 1**: Tokenizer
- Add `[` `]` tokens
- Recognize `BUKKIT` as type keyword

**Phase 2**: Parser
- Parse `identifier[expression]` as BracketAccessExpression
- Parse `identifier[expression] = value` as BracketAssignmentStatement
- Distinguish numeric vs string keys using expression type analysis

**Phase 3**: Code Generation
- Emit `identifier'Z key` for string literal keys
- Emit `identifier'Z SRS expression` for numeric keys
- Preserve operator precedence in key expressions

**Phase 4**: Testing
- Unit tests for each transpilation rule
- Integration tests with lol-coffee or LCI
- Refactor game.lol portions as proof-of-concept

## LULCODE Syntax Specification (Formal Grammar)

### BUKKIT Declaration

```ebnf
BucketDeclaration = "I HAS A" identifier "ITZ A BUKKIT"
                  | "VAR" identifier "=" "BUKKIT"   (* sugar, EPIC-003 *)
```

### BUKKIT Access

```ebnf
BucketAccess = identifier "'Z" identifier                    (* named key *)
             | identifier "'Z SRS" expression                (* numeric key *)
             | identifier "[" string-literal "]"             (* sugar: named *)
             | identifier "[" numeric-expression "]"         (* sugar: numeric *)
```

### BUKKIT Assignment

```ebnf
BucketAssignment = identifier "HAS A" identifier "ITZ" expression   (* property create *)
                 | identifier "'Z" identifier "R" expression        (* direct assign *)
                 | identifier "[" expression "]" "=" expression     (* sugar *)
```

### BUKKIT Iteration

```ebnf
BucketIteration = "IM IN YR" label "UPPIN YR" identifier 
                  "TIL BOTH SAEM" identifier "AN LENGZ OF" identifier
                  statement*
                  "IM OUTTA YR" label
                  
                | "FOREACH" identifier "," identifier "IN" identifier   (* sugar, future *)
                  statement*
                  "END"
```

## Example: game.lol Refactored with LULCODE BUKKIT

### Original LOLCODE (String-Based Workaround)

```lolcode
I HAS A VOTES ITZ ""

BTW Adding a vote
I HAS A NAME
GIMMEH NAME
I HAS A CARD
GIMMEH CARD
VOTES R SMOOSH VOTES NAME "::" CARD "," MKAY

BTW Result: "Alice::5,Bob::8,Charlie::?,"
```

**Problems**:
- Manual string building with delimiters
- No way to check if player already voted
- Parsing required in JavaScript to extract votes
- Fragile (breaks if NAME contains `:` or `,`)

### LULCODE BUKKIT Version

```lulcode
I HAS A VOTES ITZ A BUKKIT

BTW Adding a vote
I HAS A NAME
GIMMEH NAME
I HAS A CARD
GIMMEH CARD
VOTES[NAME] = CARD

BTW Check if player voted
VOTES HAS A NAME, O RLY?
  YA RLY
    VISIBLE "Already voted!"
OIC

BTW Get specific vote
VISIBLE SMOOSH "Alice voted: " VOTES["Alice"] MKAY
```

**Improvements**:
- Direct dictionary access
- Key existence check built-in
- No string parsing needed
- Type-safe, prevents delimiter injection

### Transpiled LOLCODE

```lolcode
I HAS A VOTES ITZ A BUKKIT

I HAS A NAME
GIMMEH NAME
I HAS A CARD
GIMMEH CARD
VOTES'Z SRS NAME R CARD

VOTES HAS A NAME, O RLY?
  YA RLY
    VISIBLE "Already voted!"
OIC

VISIBLE SMOOSH "Alice voted: " VOTES'Z Alice MKAY
```
