# Story 002.3: String Substring/Slice Operation - Research & Design

**Status**: In Progress  
**Date**: 2026-05-19

## Overview

Design substring extraction for LULCODE to complement string interpolation (002.1). Current LOLCODE only supports single-character access via `AT`, making substring operations cumbersome.

## Current LOLCODE String Operations

### What Exists
- **Single character access**: `str AT index` (0-based)
  - Returns single character at position
  - Example: `"HELLO" AT 1` → `"E"`
- **Length**: `LENGZ OF str` (lol-coffee extension)
- **Char code**: `ORDZ OF str` / `CHARZ OF code` (lol-coffee)

### What's Missing
- Substring extraction (multiple characters)
- Range-based slicing
- End-relative indexing (negative indices)

## Use Cases

### Real-world Examples
```lolcode
BTW Extract filename from path
I HAS A path ITZ "/home/user/file.txt"
BTW Want: "file.txt" (characters after last "/")
BTW Current: Must loop character-by-character with AT

BTW Parse wire protocol message
I HAS A msg ITZ "CMD:JOIN:Alice"
BTW Want: "Alice" (after second colon)
BTW Current: Manual parsing with AT in loop

BTW Truncate for display
I HAS A title ITZ "Very Long Title That Needs Truncation"
BTW Want: First 20 characters + "..."
BTW Current: Loop through first 20 chars
```

## Syntax Design Options

### Option 1: Verbose English-like (LOLCODE style)
```lolcode
SUBSTRING str FROM start TO end
SUBSTRING str FROM start LENGZ count

BTW Examples
I HAS A sub ITZ SUBSTRING "HELLO WORLD" FROM 0 TO 5
VISIBLE sub  BTW "HELLO"

I HAS A sub2 ITZ SUBSTRING "HELLO WORLD" FROM 6 LENGZ 5
VISIBLE sub2  BTW "WORLD"
```

**Pros**:
- Very LOLCODE-y (verbose, English-like)
- `FROM...TO` reads naturally
- `FROM...LENGZ` useful for fixed-length extraction

**Cons**:
- Very verbose for common operation
- Two variants (TO vs LENGZ) add complexity
- Doesn't fit existing operator patterns

### Option 2: Property-style (BUKKIT-like)
```lolcode
str'Z SLICE start end
str'Z SLICE start LENGZ count

BTW Examples
I HAS A sub ITZ "HELLO WORLD"'Z SLICE 0 5
VISIBLE sub  BTW "HELLO"

I HAS A sub2 ITZ "HELLO WORLD"'Z SLICE 6 LENGZ 5
VISIBLE sub2  BTW "WORLD"
```

**Pros**:
- Consistent with `str'Z LENGZ` pattern
- More concise
- Feels like a string "method"

**Cons**:
- `'Z` syntax already cryptic
- LENGZ variant still complex

### Option 3: Extended AT Operator
```lolcode
str AT start THRU end
str AT start LENGZ count

BTW Examples
I HAS A sub ITZ "HELLO WORLD" AT 0 THRU 4
VISIBLE sub  BTW "HELLO"

I HAS A sub2 ITZ "HELLO WORLD" AT 6 LENGZ 5
VISIBLE sub2  BTW "WORLD"
```

**Pros**:
- Extends existing `AT` operator naturally
- `THRU` is LOLCODE-y (used in loops: `TIL`)
- Less syntax to learn

**Cons**:
- Overloads `AT` meaning (single char vs range)
- LENGZ conflicts with `LENGZ OF` operator

### Option 4: Python/JavaScript-style SLICE
```lolcode
SLICE str FROM start TO end

BTW Examples
I HAS A sub ITZ SLICE "HELLO WORLD" FROM 0 TO 5
VISIBLE sub  BTW "HELLO"

I HAS A sub2 ITZ SLICE "HELLO WORLD" FROM 6 TO 11
VISIBLE sub2  BTW "WORLD"
```

**Pros**:
- Single clear operation name
- `FROM...TO` familiar to programmers
- No overloading of existing operators

**Cons**:
- Still somewhat verbose
- No start+length variant

### Option 5: LULCODE Sugar (brackets for ranges)
```lulcode
BTW LULCODE-only syntax (transpiles to LOLCODE)
str[start:end]
str[start:start+length]

BTW Examples
I HAS A sub ITZ "HELLO WORLD"[0:5]
VISIBLE sub  BTW "HELLO"

I HAS A sub2 ITZ "HELLO WORLD"[6:11]
VISIBLE sub2  BTW "WORLD"
```

**Pros**:
- Very concise, familiar to Python/JavaScript programmers
- Consistent with BUKKIT bracket sugar: `arr[key]`
- Can support negative indices: `str[-5:]` for last 5 chars

**Cons**:
- Not valid LOLCODE (LULCODE-only)
- Requires parser to distinguish `arr[key]` vs `str[range]`
- Colon syntax not LOLCODE-y

## Recommendation

### Choice: Hybrid Approach

**LULCODE Sugar** (transpiles):
```lulcode
str[start:end]        → transpiles to...
```

**Valid LOLCODE** (transpilation target):
```lolcode
SLICE str FROM start TO end
```

**Rationale**:
1. **LULCODE users** get concise Python-style syntax
2. **Backward compatible**: `SLICE` can be added to LOLCODE spec later
3. **Transpilation**: Generate loop-based extraction using `AT`
4. **Consistent**: Matches BUKKIT bracket sugar philosophy

### Syntax Specification

#### LULCODE Syntax
```lulcode
BTW Basic slice: start (inclusive) to end (exclusive)
str[start:end]

BTW From start to end of string
str[start:]

BTW From beginning to end
str[:end]

BTW Negative indices (end-relative)
str[-5:]     BTW Last 5 characters
str[:-5]     BTW All but last 5 characters

BTW Full copy
str[:]
```

#### LOLCODE Transpilation Target
```lolcode
SLICE str FROM start TO end

BTW Open-ended variants
SLICE str FROM start         BTW to end
SLICE str TO end             BTW from beginning
```

## Transpilation Strategy

### Challenge
LCI/LOLCODE has no built-in substring operation. Must transpile to character-by-character extraction.

### Approach: Generate Helper Function

For each use of slice, generate a LOLCODE loop:

**LULCODE Input**:
```lulcode
I HAS A greeting ITZ "HELLO WORLD"
I HAS A word ITZ greeting[0:5]
```

**LOLCODE Output**:
```lolcode
I HAS A greeting ITZ "HELLO WORLD"
BTW === LULCODE: greeting[0:5] ===
I HAS A __slice_result ITZ ""
I HAS A __slice_i ITZ 0
IM IN YR __slice_loop UPPIN YR __slice_i TIL BOTH SAEM __slice_i AN 5
  I HAS A __slice_char ITZ greeting AT __slice_i
  __slice_result R SMOOSH __slice_result AN __slice_char MKAY
IM OUTTA YR __slice_loop
I HAS A word ITZ __slice_result
```

### Optimization: Runtime Library

Alternatively, generate a reusable function once per file:

```lolcode
BTW === LULCODE Runtime: String Slice ===
HOW IZ I __LULCODE_SLICE YR str AN YR start AN YR end
  I HAS A result ITZ ""
  I HAS A i ITZ start
  IM IN YR loop UPPIN YR i TIL BOTH SAEM i AN end
    I HAS A char ITZ str AT i
    result R SMOOSH result AN char MKAY
  IM OUTTA YR loop
  FOUND YR result
IF U SAY SO

BTW === User Code ===
I HAS A greeting ITZ "HELLO WORLD"
I HAS A word ITZ I IZ __LULCODE_SLICE YR greeting AN YR 0 AN YR 5 MKAY
```

**Trade-offs**:
- **Inline**: Simpler transpilation, larger output
- **Function**: DRY, smaller output, requires function support

**Decision**: Start with **inline** for MVP, optimize to **function** in later story.

## Edge Cases

### Negative Indices
```lulcode
str[-5:]     BTW Last 5 chars
str[:-5]     BTW All but last 5
str[-8:-3]   BTW Range from end
```

**Transpilation**:
```lolcode
BTW Convert negative to positive: index + LENGZ OF str
I HAS A __str_len ITZ LENGZ OF str
I HAS A __actual_start ITZ SUM OF __str_len AN -5
```

### Out of Bounds
```lulcode
"HELLO"[0:100]   BTW Should return "HELLO", not error
"HELLO"[10:20]   BTW Should return ""
```

**Behavior**: Clamp to string bounds (Python-style)

### Invalid Ranges
```lulcode
"HELLO"[5:2]     BTW Start > end, should return ""
```

**Behavior**: Return empty string

## Implementation Plan

### Phase 1: Parser Enhancement
- Detect `identifier[start:end]` pattern
- Parse start/end expressions (can be variables, literals, or expressions)
- Handle optional start/end: `[:]`, `[5:]`, `[:10]`

### Phase 2: Negative Index Handling
- Detect negative literals: `[-5:]`
- Generate code to convert to positive indices

### Phase 3: Code Generation
- Generate inline loop for slice operation
- Handle edge cases (bounds checking)
- Optimize for literal indices where possible

### Phase 4: Testing
- Test all slice variants
- Test negative indices
- Test edge cases
- Benchmark performance (inline vs function approach)
