# Story 003.1: Variable Declaration Shorthand - Research & Design

**Status**: In Progress  
**Date**: 2026-05-19

## Overview

Design shorter variable declaration syntax to reduce LOLCODE's verbosity. Current syntax requires `I HAS A name` (4 tokens) for every variable, making code unnecessarily long.

## Current LOLCODE Variable Syntax

### Declaration
```lolcode
I HAS A name                    BTW Declare (defaults to NOOB)
I HAS A age ITZ 30              BTW Declare + initialize
I HAS A count ITZ A NUMBR       BTW Declare with type (still NOOB)
```

### Assignment
```lolcode
name R "Alice"                  BTW Assign to existing variable
age R SUM OF age AN 1           BTW Update value
```

### Type Casting
```lolcode
age IS NOW A YARN               BTW Cast variable to type
MAEK age A NUMBR                BTW Cast expression
```

## Problem: Verbosity

From `game.lol` (real-world example):
```lolcode
I HAS A NAME
I HAS A CARD
I HAS A RTXT ITZ "NO"
I HAS A VOTES ITZ 0
I HAS A PLAYER_NAME
I HAS A PLAYER_RTXT
I HAS A PLAYER_VOTES
I HAS A PLAYER_STATUS
I HAS A i ITZ 0
I HAS A ACTION
I HAS A TARGET
I HAS A CMD
I HAS A VERB
I HAS A OBJ1
I HAS A OBJ2
I HAS A OBJ3
I HAS A RESULT
```

**17 declarations** = **68 tokens** just for `I HAS A`!

## Design Goals

1. **Reduce verbosity**: Shorter syntax for common case
2. **Maintain clarity**: Still readable, not cryptic
3. **Backward compatible**: `I HAS A` still works
4. **Preserve LOLCODE charm**: Not too generic/boring
5. **Practical**: Competitive with mainstream languages

## Syntax Design Options

### Option 1: VAR Keyword (JavaScript-inspired)
```lulcode
VAR name                        BTW Simple declaration
VAR age ITZ 30                  BTW Declare + initialize
VAR count = 30                  BTW Alt syntax with =
VAR x, y, z                     BTW Multiple declarations
VAR a = 1, b = 2                BTW Multiple with init
```

**Pros**:
- Familiar to JavaScript/Go programmers
- VAR is 3 chars vs "I HAS A" (8 chars) = 63% reduction
- Multiple declarations possible
- Works with `=` operator (if we add it)

**Cons**:
- Generic, loses some LOLCODE charm
- "VAR" not lolcat-speak

**Transpilation**:
```lolcode
VAR name                → I HAS A name
VAR age ITZ 30          → I HAS A age ITZ 30
VAR x, y, z             → I HAS A x
                          I HAS A y
                          I HAS A z
```

### Option 2: LET Keyword (Lua/Swift-inspired)
```lulcode
LET name
LET age ITZ 30
LET x, y, z
```

**Pros**:
- Even shorter than VAR (3 chars)
- Familiar to Rust/Swift programmers
- LET has imperative feel

**Cons**:
- Same issues as VAR (generic, not lolcat-y)
- Conflicts with potential "immutable" meaning (Rust/Swift)

### Option 3: HAS Keyword (Shortened LOLCODE)
```lulcode
HAS name
HAS age ITZ 30
HAS x, y, z
```

**Pros**:
- Directly derived from LOLCODE's "I HAS A"
- Preserves some LOLCODE flavor
- Clear meaning (possession)

**Cons**:
- "HAS" sounds incomplete without subject
- Not much shorter (3 chars vs 8)
- Still 5 chars saved

### Option 4: DECL Keyword (Explicit)
```lulcode
DECL name
DECL age ITZ 30
DECL x, y, z
```

**Pros**:
- Explicit "declare" meaning
- Common in older languages

**Cons**:
- Verbose (4 chars)
- Not LOLCODE-y
- Boring

### Option 5: CAN HAS (LOLCODE-inspired)
```lulcode
CAN HAS name
CAN HAS age ITZ 30
CAN HAS x, y, z
```

**Pros**:
- Pure lolcat-speak ("I can has cheezburger?")
- Preserves LOLCODE charm completely
- Culturally appropriate

**Cons**:
- Still 7 chars (only 1 saved vs "I HAS A")
- Not a significant improvement in verbosity
- "CAN HAS" is a question, not a statement

### Option 6: Hybrid Approach
Allow multiple keywords:
```lulcode
VAR name               BTW Primary (short, practical)
LET name               BTW Alternative
I HAS A name           BTW Original LOLCODE (backward compat)
```

**Pros**:
- User choice (practical vs whimsical)
- Maximum backward compatibility
- Easy to transpile (all → `I HAS A`)

**Cons**:
- More syntax to learn
- Inconsistency in codebases

## Recommendation

### Choice: **VAR** with **I HAS A** backward compat

**Rationale**:
1. **Practical**: VAR is widely recognized, 63% shorter
2. **Clear**: Unambiguous declaration keyword
3. **Flexible**: Works with multiple patterns (single, multiple, with init)
4. **Compatible**: `I HAS A` still works (transpiler passes through)
5. **Balance**: Practical enough for real work, can still use LOLCODE syntax when desired

### Syntax Specification

#### Single Declaration
```lulcode
VAR name
VAR age ITZ 30
VAR count = 30         BTW If we add = operator
```

Transpiles to:
```lolcode
I HAS A name
I HAS A age ITZ 30
I HAS A count ITZ 30
```

#### Multiple Declarations (Same Line)
```lulcode
VAR x, y, z
VAR a = 1, b = 2, c = 3
```

Transpiles to:
```lolcode
I HAS A x
I HAS A y
I HAS A z
I HAS A a ITZ 1
I HAS A b ITZ 2
I HAS A c ITZ 3
```

#### With Type Hints (Future)
```lulcode
VAR count:NUMBR ITZ 0
VAR name:YARN
```

Transpiles to:
```lolcode
I HAS A count ITZ 0
count IS NOW A NUMBR
I HAS A name
name IS NOW A YARN
```

#### Backward Compatibility
```lolcode
I HAS A original ITZ "LOLCODE"    BTW Still works
VAR modern ITZ "LULCODE"          BTW Also works
```

Both pass through transpiler (or VAR → I HAS A)

## Transpilation Strategy

### Simple Pattern Matching (MVP)
```javascript
// Pattern 1: VAR name
output = output.replace(/\bVAR\s+(\w+)\b/g, 'I HAS A $1');

// Pattern 2: VAR name ITZ value
output = output.replace(/\bVAR\s+(\w+)\s+ITZ\s+(.+)/g, 'I HAS A $1 ITZ $2');

// Pattern 3: VAR name = value (if = operator added)
output = output.replace(/\bVAR\s+(\w+)\s*=\s*(.+)/g, 'I HAS A $1 ITZ $2');

// Pattern 4: VAR x, y, z (multiple declarations)
output = output.replace(/\bVAR\s+((?:\w+(?:\s*,\s*)?)+)/g, (match, vars) => {
  return vars.split(',').map(v => `I HAS A ${v.trim()}`).join('\n');
});
```

### Edge Cases

#### Multiple with Mixed Init
```lulcode
VAR a, b ITZ 10, c
```

**Ambiguity**: Does `ITZ 10` apply to `b` only or `a, b`?

**Decision**: Only last variable can have initializer in comma list:
```lulcode
VAR a, b, c ITZ 10     BTW OK: only c = 10
VAR a ITZ 1, b, c      BTW ERROR: initializer must be last
VAR a ITZ 1, b ITZ 2   BTW ERROR: can't mix in comma list
```

Better: Use separate lines for initialized vars:
```lulcode
VAR a, b, c            BTW All NOOB
c ITZ 10               BTW Initialize c after
```

Or: One VAR per initialized variable:
```lulcode
VAR a ITZ 1
VAR b ITZ 2
VAR c ITZ 3
```

#### In BUKKIT Declaration
```lulcode
VAR player ITZ A BUKKIT
```

Transpiles to:
```lolcode
I HAS A player ITZ A BUKKIT
```

#### In Function Parameters
Function parameters don't use declarations:
```lolcode
HOW IZ I greet YR name    BTW name is implicit parameter
  VISIBLE "Hello :{name}"
IF U SAY SO
```

No change needed - VAR not used in function signatures.

## Implementation Impact

### Before (game.lol excerpt)
```lolcode
I HAS A NAME
I HAS A CARD
I HAS A RTXT ITZ "NO"
I HAS A VOTES ITZ 0
I HAS A PLAYER_NAME
I HAS A PLAYER_RTXT
I HAS A PLAYER_VOTES
I HAS A PLAYER_STATUS
```

**8 lines, 40 tokens**

### After (LULCODE)
```lulcode
VAR NAME, CARD
VAR RTXT ITZ "NO"
VAR VOTES ITZ 0
VAR PLAYER_NAME, PLAYER_RTXT, PLAYER_VOTES, PLAYER_STATUS
```

**4 lines, 18 tokens** (55% reduction!)

Or more readable:
```lulcode
VAR NAME, CARD
VAR RTXT ITZ "NO", VOTES ITZ 0
VAR PLAYER_NAME, PLAYER_RTXT, PLAYER_VOTES, PLAYER_STATUS
```

**3 lines, 16 tokens** (60% reduction!)

## Examples

### Example 1: Simple Variables
```lulcode
HAI 1.2
VAR x, y, z
x ITZ 10
y ITZ 20
z ITZ SUM OF x AN y
VISIBLE "Result: {z}"
KTHXBYE
```

Transpiles to:
```lolcode
HAI 1.2
I HAS A x
I HAS A y
I HAS A z
x R 10
y R 20
z R SUM OF x AN y
VISIBLE "Result: :{z}"
KTHXBYE
```

### Example 2: With Initialization
```lulcode
HAI 1.2
VAR name ITZ "Alice"
VAR age ITZ 30
VAR greeting ITZ "Hello {name}!"
VISIBLE greeting
KTHXBYE
```

### Example 3: BUKKIT with VAR
```lulcode
HAI 1.2
VAR player ITZ A BUKKIT
player HAS A name ITZ "Bob"
player HAS A score ITZ 100

VAR pName ITZ player'Z name
VISIBLE "Player: {pName}"
KTHXBYE
```

## Testing Strategy

1. **Test single declaration**: `VAR x` → `I HAS A x`
2. **Test with init**: `VAR x ITZ 5` → `I HAS A x ITZ 5`
3. **Test multiple**: `VAR x, y, z` → three `I HAS A` statements
4. **Test mixed**: `VAR x, y ITZ 10`
5. **Test backward compat**: `I HAS A x` unchanged
6. **Test with BUKKIT**: `VAR arr ITZ A BUKKIT`
7. **Test in real program**: Refactor game.lol excerpt
