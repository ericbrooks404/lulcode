# Story 003.2: Conditional Statement Shorthand - Research & Design

**Status**: In Progress  
**Date**: 2026-05-19

## Overview

Design cleaner conditional syntax to reduce LOLCODE's notorious verbosity. Current conditionals require complex expressions and verbose keywords, making simple comparisons painful.

## Current LOLCODE Conditional Syntax

### Basic IF Statement
```lolcode
BOTH SAEM x AN y, O RLY?
  YA RLY
    VISIBLE "Equal"
  NO WAI
    VISIBLE "Not equal"
OIC
```

**7+ tokens** for a simple equality check!

### With Expression Evaluation
```lolcode
BOTH SAEM VERB AN "LOCAL-JOIN", O RLY?
  YA RLY
    BTW Handle join
OIC
```

### Multiple Conditions (MEBBE)
```lolcode
BOTH SAEM x AN 1, O RLY?
  YA RLY
    VISIBLE "One"
  MEBBE BOTH SAEM x AN 2
    VISIBLE "Two"
  NO WAI
    VISIBLE "Other"
OIC
```

### Nested Conditionals
```lolcode
BOTH SAEM action AN "VOTE", O RLY?
  YA RLY
    BOTH SAEM target AN IT, O RLY?
      YA RLY
        BTW Vote for self
    OIC
OIC
```

## Problem: Extreme Verbosity

From `game.lol`:
- 16+ instances of `BOTH SAEM...O RLY?` patterns
- Deeply nested conditionals hard to read
- Prefix comparison operators awkward
- `O RLY?`, `YA RLY`, `NO WAI`, `OIC` add visual noise

## Design Goals

1. **Reduce verbosity**: Shorter syntax for common cases
2. **Infix operators**: `x == y` more natural than `BOTH SAEM x AN y`
3. **Clear structure**: Easy to see condition vs body
4. **Preserve option**: Keep LOLCODE syntax available
5. **Maintain charm**: Don't make it too generic

## Part 1: Comparison Operators

### Current LOLCODE Operators
```lolcode
BOTH SAEM x AN y          BTW Equal
DIFFRINT x AN y           BTW Not equal
BIGGR OF x AN y           BTW Greater (returns max)
SMALLR OF x AN y          BTW Lesser (returns min)
```

Note: BIGGR/SMALLR return values, not booleans! Must compare:
```lolcode
BOTH SAEM x AN BIGGR OF x AN y    BTW x >= y
```

### Option 1: Standard Infix Operators
```lulcode
x == y                    BTW Equal
x != y                    BTW Not equal
x > y                     BTW Greater than
x < y                     BTW Less than
x >= y                    BTW Greater or equal
x <= y                    BTW Less or equal
```

**Pros**:
- Universal syntax, everyone knows it
- Most concise
- Natural left-to-right reading

**Cons**:
- Completely generic, loses LOLCODE character
- Not words (LOLCODE is word-based)

### Option 2: Word-Based Infix
```lulcode
x IS y                    BTW Equal
x ISNT y                  BTW Not equal
x BIGR y                  BTW Greater
x SMALR y                 BTW Less
```

**Pros**:
- Still readable as words
- Preserves some LOLCODE flavor
- Shorter than prefix operators

**Cons**:
- Non-standard (confusing to newcomers)
- "IS" ambiguous (assignment? comparison?)

### Option 3: Shortened Prefix
```lulcode
SAME x y                  BTW Equal
DIFF x y                  BTW Not equal
BIGR x y                  BTW Greater
SMALR x y                 BTW Less
```

**Pros**:
- Keeps prefix style (consistent with LOLCODE)
- Shorter than BOTH SAEM / DIFFRINT
- Clear meaning

**Cons**:
- Still prefix (less natural for many)
- Not much shorter

### Recommendation: **Hybrid Approach**

Support both for flexibility:

**Primary (infix operators)**:
```lulcode
x == y
x != y
x > y
x < y
x >= y
x <= y
```

**Backward compatible (prefix)**:
```lolcode
BOTH SAEM x AN y
DIFFRINT x AN y
```

**Transpilation**: All → LOLCODE prefix form

## Part 2: Conditional Statements

### Option 1: IF...END (Lua/Ruby-style)
```lulcode
IF x == y
  VISIBLE "Equal"
END

IF x == y
  VISIBLE "Equal"
ELSE
  VISIBLE "Not equal"
END

IF x == 1
  VISIBLE "One"
ELIF x == 2
  VISIBLE "Two"
ELSE
  VISIBLE "Other"
END
```

**Pros**:
- Clean, minimal syntax
- Clear structure (IF/ELSE/END)
- ELIF for else-if
- Most compact option

**Cons**:
- Very generic (Python/Lua/Ruby have this)
- Loses LOLCODE whimsy completely

### Option 2: IF...THEN...KTHX (LOLCODE-inspired)
```lulcode
IF x == y THEN
  VISIBLE "Equal"
KTHX

IF x == y THEN
  VISIBLE "Equal"
NOWAI
  VISIBLE "Not equal"
KTHX
```

**Pros**:
- KTHX preserves LOLCODE flavor ("k thx")
- NOWAI familiar to LOLCODE users
- THEN makes structure clear

**Cons**:
- THEN adds extra token (vs IF condition directly)
- NOWAI vs ELSE inconsistent

### Option 3: O HAI...KTHX (New LOLCODE-y)
```lulcode
IF x == y, O HAI
  VISIBLE "Equal"
KTHX

IF x == y, O HAI
  VISIBLE "Equal"
NOWAI
  VISIBLE "Not equal"
KTHX
```

**Pros**:
- O HAI is lolcat-speak greeting
- Very LOLCODE-y
- KTHX is "k thx bye" 

**Cons**:
- Comma after condition awkward
- O HAI usually means "hello" not "then"

### Option 4: Keep O RLY? with Infix
```lulcode
x == y, O RLY?
  YA RLY
    VISIBLE "Equal"
  NO WAI
    VISIBLE "Not equal"
OIC
```

**Pros**:
- Preserves LOLCODE conditional structure
- Only replaces prefix operators with infix
- Familiar to LOLCODE users

**Cons**:
- Still verbose (O RLY?, YA RLY, NO WAI, OIC)
- Minimal improvement

### Recommendation: **IF...END with LOLCODE alternatives**

**Primary (clean)**:
```lulcode
IF condition
  BTW body
END

IF condition
  BTW if body
ELSE
  BTW else body
END

IF condition1
  BTW body 1
ELIF condition2
  BTW body 2
ELSE
  BTW else body
END
```

**Backward compatible**:
```lolcode
condition, O RLY?
  YA RLY
    BTW body
OIC
```

**Rationale**:
- IF...END is universal, practical, minimal
- ELIF clearer than MEBBE
- Can still use O RLY? for LOLCODE flavor
- Biggest verbosity reduction

## Part 3: Logical Operators

### Current LOLCODE
```lolcode
BOTH OF x AN y            BTW AND (both truthy)
EITHER OF x AN y          BTW OR (either truthy)
WON OF x AN y             BTW XOR (exactly one truthy)
NOT x                     BTW NOT
ALL OF x AN y AN z MKAY   BTW N-ary AND
ANY OF x AN y AN z MKAY   BTW N-ary OR
```

### Option 1: Standard Infix
```lulcode
x && y                    BTW AND
x || y                    BTW OR
!x                        BTW NOT
x && y && z               BTW Multiple AND
```

**Pros**:
- Universal, everyone knows it
- Most concise
- Chainable naturally

**Cons**:
- Symbolic, not words
- Loses LOLCODE character

### Option 2: Word-Based Infix
```lulcode
x AND y
x OR y
NOT x
x AND y AND z
```

**Pros**:
- Readable as English
- Clear meaning
- Still word-based (LOLCODE-ish)

**Cons**:
- Longer than symbols
- Very generic

### Recommendation: **Support Both**

```lulcode
x && y                    BTW Symbolic (short)
x AND y                   BTW Word-based (readable)
!x                        BTW Symbolic NOT
NOT x                     BTW Word-based NOT
```

All transpile to:
```lolcode
BOTH OF x AN y
NOT x
```

## Complete Syntax Specification

### Infix Comparison Operators
```lulcode
==    →  BOTH SAEM x AN y
!=    →  DIFFRINT x AN y
>     →  BOTH SAEM x AN BIGGR OF x AN y (plus NOT equal check)
<     →  BOTH SAEM y AN BIGGR OF x AN y (plus NOT equal check)
>=    →  BOTH SAEM x AN BIGGR OF x AN y
<=    →  BOTH SAEM y AN BIGGR OF x AN y
```

Note: `>` and `<` need special handling since BIGGR/SMALLR return max/min.

### Logical Operators
```lulcode
&&, AND       →  BOTH OF x AN y
||, OR        →  EITHER OF x AN y
!, NOT        →  NOT x
```

### Conditional Statements
```lulcode
IF condition
  body
END

IF condition
  if-body
ELSE
  else-body
END

IF condition1
  body1
ELIF condition2
  body2
ELSE
  else-body
END
```

All transpile to:
```lolcode
condition, O RLY?
  YA RLY
    body
  NO WAI / MEBBE
    ...
OIC
```

## Transpilation Strategy

### Step 1: Convert Infix Operators
```javascript
// == operator
output = output.replace(/(\w+)\s*==\s*(\w+)/g, 'BOTH SAEM $1 AN $2');

// != operator
output = output.replace(/(\w+)\s*!=\s*(\w+)/g, 'DIFFRINT $1 AN $2');

// >= operator
output = output.replace(/(\w+)\s*>=\s*(\w+)/g, 
  'BOTH SAEM $1 AN BIGGR OF $1 AN $2');

// && operator
output = output.replace(/(\S+)\s*&&\s*(\S+)/g, 'BOTH OF $1 AN $2');
```

### Step 2: Convert IF Statements
```javascript
// IF...END (no else)
output = output.replace(
  /IF\s+(.+?)\n([\s\S]+?)END/g,
  (match, condition, body) => {
    return `${condition}, O RLY?\n  YA RLY\n${indent(body)}\nOIC`;
  }
);

// IF...ELSE...END
output = output.replace(
  /IF\s+(.+?)\n([\s\S]+?)ELSE\n([\s\S]+?)END/g,
  (match, condition, ifBody, elseBody) => {
    return `${condition}, O RLY?\n  YA RLY\n${indent(ifBody)}  NO WAI\n${indent(elseBody)}OIC`;
  }
);
```

### Step 3: Handle ELIF
```javascript
// Convert ELIF to MEBBE
output = output.replace(/ELIF\s+(.+?)\n/g, 'MEBBE $1\n');
```

## Examples

### Example 1: Simple Comparison
```lulcode
HAI 1.2
VAR x ITZ 5, y ITZ 5

IF x == y
  VISIBLE "Equal!"
END

KTHXBYE
```

Transpiles to:
```lolcode
HAI 1.2
I HAS A x ITZ 5
I HAS A y ITZ 5

BOTH SAEM x AN y, O RLY?
  YA RLY
    VISIBLE "Equal!"
OIC

KTHXBYE
```

### Example 2: IF-ELSE
```lulcode
HAI 1.2
VAR age ITZ 25

IF age >= 18
  VISIBLE "Adult"
ELSE
  VISIBLE "Minor"
END

KTHXBYE
```

Transpiles to:
```lolcode
HAI 1.2
I HAS A age ITZ 25

BOTH SAEM age AN BIGGR OF age AN 18, O RLY?
  YA RLY
    VISIBLE "Adult"
  NO WAI
    VISIBLE "Minor"
OIC

KTHXBYE
```

### Example 3: Multiple Conditions (ELIF)
```lulcode
HAI 1.2
VAR score ITZ 85

IF score >= 90
  VISIBLE "A"
ELIF score >= 80
  VISIBLE "B"
ELIF score >= 70
  VISIBLE "C"
ELSE
  VISIBLE "F"
END

KTHXBYE
```

### Example 4: Logical Operators
```lulcode
HAI 1.2
VAR x ITZ 5, y ITZ 10

IF x > 0 && y > 0
  VISIBLE "Both positive"
END

IF x == 5 OR y == 5
  VISIBLE "One is five"
END

KTHXBYE
```

### Example 5: Complex Condition
```lulcode
HAI 1.2
VAR action ITZ "VOTE", target ITZ "Alice"

IF action == "VOTE" AND target != IT
  VISIBLE "Voting for someone else"
ELSE
  VISIBLE "Self-vote or different action"
END

KTHXBYE
```

## Edge Cases

### Operator Precedence
```lulcode
x == 5 && y > 10          BTW Which to evaluate first?
```

**Solution**: Use parentheses for complex conditions (defer to future)
```lulcode
(x == 5) && (y > 10)      BTW Clear precedence
```

Or: Transpile left-to-right with explicit grouping.

### Multi-line Conditions
```lulcode
IF x == 5 &&
   y == 10
  VISIBLE "Match"
END
```

**Solution**: Support line continuation (defer or require single line).

### Nested IFs
```lulcode
IF x > 0
  IF y > 0
    VISIBLE "Both positive"
  END
END
```

Should work naturally with END pairing.

## Impact Analysis

### Before (game.lol excerpt)
```lolcode
BOTH SAEM VERB AN "LOCAL-JOIN", O RLY?
  YA RLY
    BTW Handle join
    BOTH SAEM TARGET AN IT, O RLY?
      YA RLY
        BTW Error case
      NO WAI
        BTW Normal case
    OIC
OIC
```

**Lines**: 10  
**Tokens**: ~30

### After (LULCODE)
```lulcode
IF VERB == "LOCAL-JOIN"
  BTW Handle join
  IF TARGET != IT
    BTW Normal case
  ELSE
    BTW Error case
  END
END
```

**Lines**: 8  
**Tokens**: ~15

**50% reduction!**
