# Story 003.3: Assignment Operator Shorthand - Research & Design

**Status**: In Progress  
**Date**: 2026-05-19

## Overview

Design shorter assignment syntax. Current LOLCODE uses `R` ("are") which is cute but non-standard. Adding `=` as an alternative makes LULCODE more accessible.

## Current LOLCODE Assignment Syntax

### Basic Assignment
```lolcode
x R 5                     BTW "x are 5"
name R "Alice"
result R SUM OF x AN y
```

### Property Assignment (BUKKIT)
```lolcode
player'Z score R 100
arr'Z key R "value"
```

### Cast Assignment
```lolcode
x IS NOW A NUMBR          BTW Type cast + assign
```

## Design Goals

1. **Familiar syntax**: `=` is universal
2. **Keep R**: Backward compatibility
3. **Simple transpilation**: `=` → `R`
4. **Compound operators**: `+=`, `-=`, etc. (future)

## Syntax Design Options

### Option 1: Add = Operator (Recommended)
```lulcode
x = 5                     BTW Clean, universal
name = "Alice"
result = SUM OF x AN y
x R 5                     BTW Still works (backward compat)
```

**Pros**:
- Universal syntax, everyone knows `=`
- No learning curve
- Most concise

**Cons**:
- Generic, loses LOLCODE character
- Two ways to do same thing (inconsistency)

**Transpilation**:
```lolcode
x = 5  →  x R 5
```

### Option 2: Keep R Only
```lulcode
x R 5                     BTW Original LOLCODE
```

**Pros**:
- Maintains LOLCODE character
- Consistent (only one way)
- No changes needed

**Cons**:
- Unfamiliar to programmers
- Awkward to type
- Not actually shorter

### Option 3: IZ (Alternative LOLCODE-y)
```lulcode
x IZ 5                    BTW "x is 5"
name IZ "Alice"
```

**Pros**:
- LOLCODE-flavored
- Natural English reading
- Shorter than `R`

**Cons**:
- Conflicts with `IZ` used in function calls: `I IZ function`
- Ambiguous

### Recommendation: **Support Both = and R**

```lulcode
x = 5                     BTW Primary (concise)
x R 5                     BTW Backward compatible
```

**Transpilation**: Both → `x R 5`

**Rationale**:
- `=` is universal, removes barrier to entry
- Keeping `R` preserves LOLCODE charm for those who want it
- Simple transpilation (just replace `=` with `R`)
- User can choose style (practical vs whimsical)

## Compound Assignment Operators (Future)

### Standard Compound Operators
```lulcode
x += 5                    BTW x = x + 5
x -= 5                    BTW x = x - 5
x *= 5                    BTW x = x * 5
x /= 5                    BTW x = x / 5
x %= 5                    BTW x = x % 5
```

**Transpilation**:
```lolcode
x += 5  →  x R SUM OF x AN 5
x -= 5  →  x R DIFF OF x AN 5
x *= 5  →  x R PRODUKT OF x AN 5
x /= 5  →  x R QUOSHUNT OF x AN 5
x %= 5  →  x R MOD OF x AN 5
```

### String Compound
```lulcode
str += "text"             BTW Append to string
```

Transpiles to:
```lolcode
str R SMOOSH str AN "text" MKAY
```

**Decision**: Defer compound operators to future EPIC. Start with simple `=` only.

## Assignment in Expressions

LOLCODE doesn't support assignment in expressions:
```lolcode
BTW This doesn't work in LOLCODE:
IF (x R 5), O RLY?        BTW Can't assign in condition
```

LULCODE won't either (for now):
```lulcode
BTW Not supported:
IF x = 5                  BTW ERROR: = is comparison or assignment?
```

Use explicit separate assignment:
```lulcode
x = 5
IF x == 5                 BTW Clear: == is comparison
```

## Property Assignment with =

BUKKIT property assignment should work with `=`:
```lulcode
player["score"] = 100
player'Z score = 100      BTW After BUKKIT transpilation
```

Transpiles to:
```lolcode
player'Z score R 100
```

## Transpilation Strategy

### Pattern Matching
```javascript
// Simple assignment: var = value
output = output.replace(/(\w+)\s*=\s*([^=].+?)(?=\n|$)/g, '$1 R $2');

// Avoid ==, !=, >=, <= (comparison operators)
// Use negative lookahead for =
output = output.replace(/(\w+)\s*=\s+(?!=)(.+)/g, '$1 R $2');
```

### Edge Cases

#### Assignment vs Comparison
```lulcode
x = 5                     BTW Assignment (single =)
x == 5                    BTW Comparison (double ==)
```

**Solution**: Pattern must distinguish `=` from `==`, `!=`, `>=`, `<=`.

#### Multiple = on Same Line
```lulcode
x = y = 5                 BTW Chained assignment
```

**Decision**: Not supported in MVP. Require separate lines:
```lulcode
y = 5
x = y
```

#### Assignment in String
```lulcode
VISIBLE "x = 5"           BTW String literal, not assignment
```

**Solution**: Don't transform inside string literals (already handled by string interpolation processing).

## Examples

### Example 1: Simple Assignment
```lulcode
HAI 1.2
VAR x, y

x = 10
y = 20
VISIBLE "x={x}, y={y}"

KTHXBYE
```

Transpiles to:
```lolcode
HAI 1.2
I HAS A x
I HAS A y

x R 10
y R 20
VISIBLE "x=:{x}, y=:{y}"

KTHXBYE
```

### Example 2: Mixed Styles
```lulcode
HAI 1.2
VAR x, y

x = 10                    BTW Modern style
y R 20                    BTW LOLCODE style

VISIBLE "x={x}, y={y}"
KTHXBYE
```

Both work, transpile the same way.

### Example 3: BUKKIT Assignment
```lulcode
HAI 1.2
VAR player ITZ A BUKKIT

player["name"] = "Alice"
player["score"] = 100

VAR pName = player["name"]
VISIBLE "Player: {pName}"

KTHXBYE
```

### Example 4: Computed Assignment
```lulcode
HAI 1.2
VAR x = 10, y = 20

VAR sum = SUM OF x AN y
VAR diff = DIFF OF x AN y

VISIBLE "sum={sum}, diff={diff}"
KTHXBYE
```

## Testing Strategy

1. **Test simple assignment**: `x = 5` → `x R 5`
2. **Test with VAR**: `VAR x = 5` → `I HAS A x ITZ 5`
3. **Test BUKKIT**: `arr["key"] = val` → `arr'Z key R val`
4. **Test computed**: `x = SUM OF y AN z`
5. **Test backward compat**: `x R 5` unchanged
6. **Test NOT comparison**: `x == 5` stays `x == 5` (for conditional transpilation)

## Impact Analysis

### Before
```lolcode
x R 10
y R 20
result R SUM OF x AN y
player'Z score R 100
```

### After (LULCODE)
```lulcode
x = 10
y = 20
result = SUM OF x AN y
player["score"] = 100
```

**Improvement**: Familiar syntax, easier to read for mainstream programmers.

Not a huge verbosity reduction (same tokens), but significantly improves readability and accessibility.
