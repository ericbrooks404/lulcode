# Story 006.1: Array Implementation Strategy Research

**Status**: In Progress  
**Date**: 2026-05-20

## Goal

Research and choose the best approach for implementing true numeric-indexed arrays in LULCODE, considering:
1. Self-hosting requirement (transpiler needs arrays)
2. LCI compatibility
3. Implementation complexity
4. Performance

## The Problem

**Current State**:
- ✅ BUKKIT works for string keys: `arr["Alice"]`, `arr["key"]`
- ❌ BUKKIT fails for numeric keys: `arr[0]`, `arr[1]`

**LCI Limitation** (tested 2026-05-15):
```lolcode
I HAS A arr ITZ A BUKKIT
arr HAS A 0 ITZ "value"
Error: expected identifier at: 0
```

LCI's BUKKIT only accepts identifier names, not numeric literals.

## Why Arrays Are Critical

**Self-Hosting Requirements**:
The transpiler (`src/transform.js`) needs arrays for:

```javascript
const strings = [];                    // Store extracted strings
const comments = [];                   // Store extracted comments
strings.push(content);                 // Add to array
for (let i = 0; i < strings.length; i++) {
  process(strings[i]);                 // Access by numeric index
}
```

**Without arrays**: Cannot transpile the transpiler to LULCODE.

## Option Evaluation

### Option 1: New LOLCODE Type

**Approach**: Add ARRAY as new type to LOLCODE spec

**Syntax**:
```lolcode
I HAS A list ITZ AN ARRAY
list HAS A 0 ITZ "first"
list HAS A 1 ITZ "second"
VISIBLE list'Z SRS 0
```

**Pros**:
- Clean, proper arrays
- Could optimize performance
- Follows LOLCODE conventions

**Cons**:
- ❌ Requires modifying LCI (C code)
- ❌ Breaks LOLCODE 1.2 compatibility
- ❌ May not be accepted by LOLCODE community
- ❌ Blocks progress on self-hosting

**Implementation Effort**: 4-6 weeks (modify LCI, test, maintain fork)

**Verdict**: ❌ **Too invasive for bootstrap strategy**

---

### Option 2: String-Encoded BUKKIT Keys

**Approach**: Transpile array syntax to BUKKIT with encoded string keys

**Strategy**: Use special prefix to distinguish array indices from regular keys

**Syntax (LULCODE)**:
```lulcode
VAR list ITZ []
list[0] = "first"
list[1] = "second"
VISIBLE list[0]
```

**Transpiles to (LOLCODE)**:
```lolcode
I HAS A list ITZ A BUKKIT
list HAS A __0 ITZ "first"
list HAS A __1 ITZ "second"
VISIBLE list'Z __0
```

**Encoding Scheme**:
- Numeric index `n` → string key `__n` (double underscore prefix)
- Distinguishes array indices from regular keys
- Example: `arr[0]` → `arr'Z __0`

**Implementation Details**:

1. **Array Creation**:
```lulcode
VAR arr ITZ []
→ I HAS A arr ITZ A BUKKIT
```

2. **Numeric Access/Assignment**:
```lulcode
arr[0] = "value"
→ arr'Z __0 R "value"

VISIBLE arr[5]
→ VISIBLE arr'Z __5
```

3. **Variable Index**:
```lulcode
arr[i] = "value"
→ I HAS A __idx ITZ SMOOSH "__" AN i MKAY
  arr'Z SRS __idx R "value"
```

4. **Array Length**:
Track length separately:
```lulcode
arr.__length = 0    BTW metadata key
```

**Pros**:
- ✅ Works with current LCI (no modifications)
- ✅ Maintains LOLCODE 1.2 compatibility
- ✅ Can implement immediately
- ✅ Enables self-hosting progress
- ✅ Straightforward transpilation

**Cons**:
- Performance overhead (string encoding/decoding)
- Not "true" arrays (still BUKKIT underneath)
- Length tracking requires extra bookkeeping
- Mixed array/dictionary usage could be confusing

**Implementation Effort**: 1-2 weeks (transpiler changes only)

**Verdict**: ✅ **Most practical for MVP and self-hosting**

---

### Option 3: Wait for LCI Enhancement

**Approach**: Wait for LCI to add native array support

**Pros**:
- Eventually get proper arrays
- No workarounds needed

**Cons**:
- ❌ No timeline (LCI development is slow)
- ❌ Blocks self-hosting indefinitely
- ❌ May never happen
- ❌ Can't control feature set

**Verdict**: ❌ **Not viable for self-hosting goal**

---

### Option 4: Extend lol-coffee

**Approach**: Add array type to lol-coffee VM

**lol-coffee Architecture**:
```
Tokenizer → Parser → AST → VM (CoffeeScript)
```

Could add ARRAY as VM-level type with:
- `push()`, `pop()`, `shift()` operations
- Numeric indexing
- Length tracking

**Pros**:
- Full control over implementation
- Can add exactly what we need
- No LCI dependency

**Cons**:
- ❌ lol-coffee is CoffeeScript (adds complexity)
- ❌ Larger scope than LULCODE alone
- ❌ Maintenance burden
- ❌ Users need custom lol-coffee fork
- ❌ Breaks standard LOLCODE compatibility

**Implementation Effort**: 3-4 weeks (learn codebase, modify VM, test)

**Verdict**: ❌ **Too much scope for the benefit**

## Recommendation

**Choose Option 2: String-Encoded BUKKIT Keys**

### Rationale

1. **Enables Self-Hosting**: We can implement arrays in 1-2 weeks
2. **No Dependencies**: Works with current LCI, no modifications needed
3. **Maintains Compatibility**: Transpiles to valid LOLCODE 1.2
4. **Low Risk**: If it doesn't work well, can switch to Option 1 later
5. **Practical**: "Good enough" solution that moves project forward

### Implementation Strategy

**Phase 1: Basic Arrays**
- Array literals: `[]` or `[1, 2, 3]`
- Numeric access: `arr[0]`, `arr[i]`
- Assignment: `arr[0] = value`
- Length: `LENGZ OF arr` (track separately)

**Phase 2: Array Operations**
- `PUSH element TO arr`
- `POP FROM arr`
- `SHIFT FROM arr`
- Runtime library functions

**Phase 3: Advanced Features**
- Array slicing: `arr[0:5]`
- Array concatenation
- FOREACH iteration

### Encoding Details

**Key Format**: `__<index>` (double underscore + number)
- Examples: `__0`, `__1`, `__42`
- Distinguishable from user keys (unlikely to conflict)
- Easy to parse and generate

**Metadata Keys**:
- `__length`: Track array length
- `__is_array`: Flag to distinguish from plain BUKKIT

**Example Transpilation**:

```lulcode
VAR nums ITZ [10, 20, 30]
nums[1] = 25
VISIBLE nums[1]
```

Transpiles to:

```lolcode
I HAS A nums ITZ A BUKKIT
nums HAS A __length ITZ 3
nums HAS A __0 ITZ 10
nums HAS A __1 ITZ 20
nums HAS A __2 ITZ 30
nums'Z __1 R 25
VISIBLE nums'Z __1
```

## Edge Cases to Handle

1. **Negative Indices**: `arr[-1]` (last element)
   - Convert to positive: `arr[length - 1]`
   
2. **Out of Bounds**: `arr[100]` when length is 5
   - Return NOOB (null)
   - Or extend array with NOOB gaps
   
3. **Sparse Arrays**: `arr[0] = 1; arr[5] = 2`
   - Track max index separately
   - Fill gaps with NOOB
   
4. **Mixed Usage**: Using both array indices and string keys
   - Allow but document as discouraged
   - `arr[0]` and `arr["name"]` can coexist
